c'est quoi & pourquoi a tu utiliser cela et pas autre choses a la place:

Express (se prononce "éks-prèsse")
  C'est un framework web pour Node.js qui permet de créer des API et gérer les routes HTTP.
  On l'utilise pour structurer le backend et gérer les requêtes du frontend (auth, quiz).
  Pourquoi pas Fastify ? Express est le standard, possède une énorme communauté et une documentation très riche, idéal pour ce projet.

Node.js (se prononce "node j-éss")
  Environnement d'exécution JavaScript côté serveur.
  Utilisé pour faire tourner le backend Express et gérer les connexions Socket.io.
  Pourquoi pas Python ou PHP ? Utiliser JS sur toute la stack (frontend + backend) permet de partager des connaissances et d'être plus rapide en développement.

MariaDB (se prononce "maria-dé-bé")
  C'est un système de gestion de base de données relationnelle (fork de MySQL).
  Utilisée pour stocker les utilisateurs, les quiz, les questions et les réponses de manière persistante.
  Pourquoi pas MongoDB ? Les données du quiz (quizz -> questions -> réponses) sont fortement structurées et liées, le relationnel (SQL) est donc plus adapté que le NoSQL.

mysql2 (pool) (se prononce "maï-èss-kyu-èl-tou poule")
  Librairie Node.js pour communiquer avec MariaDB/MySQL. On utilise un "pool" de connexions.
  Le pool permet de réutiliser des connexions au lieu d'en créer une nouvelle à chaque requête, ce qui est beaucoup plus performant.
  On utilise la version promise-based pour pouvoir utiliser async/await.

bcrypt (se prononce "bi-kript")
  Algorithme de hachage de mots de passe.
  Utilisé pour sécuriser les mots de passe en base (on ne stocke jamais le mot de passe en clair).
  Pourquoi pas MD5 ou SHA-1 ? Bcrypt est beaucoup plus lent et inclut un "salt", ce qui le rend résistant aux attaques par force brute et par tables arc-en-ciel.

JWT (jsonwebtoken) (se prononce "jé-wa-té")
  Standard pour créer des jetons d'accès sécurisés.
  Le serveur génère un token après la connexion, que le client renvoie dans l'en-tête Authorization.
  Pourquoi pas les sessions classiques ? JWT est "stateless" (le serveur n'a pas besoin de stocker l'état de session en RAM ou DB), ce qui facilite le passage à l'échelle.

Socket.io (Backend)
  Gère les connexions WebSocket côté serveur pour le temps réel.
  C'est le "cerveau" du jeu : il gère les salons (rooms), diffuse les questions et calcule les scores.
  Pourquoi pas les WebSockets natifs ? Socket.io gère automatiquement les reconnexions, le fallback HTTP et simplifie énormément la gestion des rooms.

Middleware (se prononce "mid-èl-ouère")
  Fonction qui s'exécute entre la requête reçue et la réponse envoyée (ex: authMiddleware).
  Utilisé pour vérifier si l'utilisateur est connecté (token valide) avant d'accéder aux routes protégées.
  Pourquoi pas vérifier dans chaque route ? Le middleware permet de factoriser le code et de sécuriser plusieurs routes d'un seul coup.

dotenv (se prononce "dote-èn-v")
  Charge les variables d'environnement depuis un fichier .env.
  Utilisé pour masquer les informations sensibles (identifiants DB, secrets JWT) du code source.
  Indispensable pour la sécurité et pour changer de configuration selon l'environnement (local vs prod).

Cors (se prononce "kor-z")
  Non utilisé explicitement car le frontend est servi par le même serveur Express (express.static).
  Si le frontend était sur un autre domaine, il faudrait configurer CORS pour autoriser les requêtes.


Comment fonctionne ton stockage de données (Schéma DB) ?

Table "users"
  Stocke l'email (unique) et le password (haché). Sert pour l'authentification des hosts.

Table "quizzes"
  Stocke le titre du quiz et l'ID de l'utilisateur qui l'a créé (clé étrangère).
  Supprimer un utilisateur supprime automatiquement ses quiz (ON DELETE CASCADE).

Table "questions"
  Reliée à un quiz. Stocke le texte, le timer (en secondes), l'index de la bonne réponse et l'ordre d'affichage.

Table "reponses"
  Reliée à une question. Stocke le texte de la réponse et son index (0, 1, 2 ou 3).


Logique Socket.io côté serveur :

rooms (salons)
  On utilise socket.join(pin). Cela permet d'envoyer un message uniquement aux joueurs d'une partie précise avec io.to(pin).emit(...).

games (objet global)
  Variable stockée en mémoire vive (RAM) du serveur qui contient toutes les parties en cours.
  Structure : games[pin] = { hostSocketId, players, currentQuiz, status, ... }.
  Pourquoi pas en base de données ? Le jeu est éphémère et nécessite des lectures/écritures ultra-rapides pour le temps réel.

generatePIN()
  Génère un code à 6 chiffres aléatoire et vérifie qu'il n'est pas déjà utilisé par une autre partie.

Calcul des points (Score logic)
  Basé sur la rapidité. On compare Date.now() avec questionStartTime.
  Plus le joueur répond vite, plus il approche des 1000 points. S'il répond à la dernière seconde, il gagne très peu de points.


Gestion de la sécurité :

Auth Rate Limit
  On limite le nombre de tentatives de connexion/inscription par IP (10 max toutes les 15 min) pour éviter les attaques par force brute.

Validation Robuste (server.js)
  Avant d'insérer un quiz en base, on vérifie TOUT : type des données, longueur des textes (max 500 pour les questions), nombre de questions (max 50), etc.
  Cela évite les injections de données malveillantes ou le remplissage inutile de la DB.

Sanitization
  On utilise .trim() sur les entrées textuelles pour enlever les espaces inutiles.
  On utilise des requêtes préparées (pool.execute("...", [params])) pour se protéger des injections SQL.
