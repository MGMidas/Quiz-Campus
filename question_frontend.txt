c'est quoi & pourquoi a tu utiliser cela et pas autre choses a la place:

fetch (se prononce "fèche")
  C'est une API JavaScript native pour faire des requêtes HTTP (GET, POST, PUT, DELETE).
  On l'utilise pour communiquer avec le serveur (API REST) : connexion, inscription, CRUD des quiz.
  Pourquoi pas axios ? Parce que fetch est natif, pas de dépendance npm, plus léger. On aurait pu utiliser XMLHttpRequest mais fetch est plus moderne et lisible.

toasts (se prononce "tost")
  Ce sont des notifications temporaires qui apparaissent en haut de l'écran (succès, erreur).
  Pourquoi pas une popup/alert ? Les toasts sont non-bloquants, ne coupent pas l'action, meilleure UX.
  On les a implémentés avec une div #toast + classes CSS (.visible, .success, .error) — pas de librairie externe.

sessionStorage (se prononce "séssion-storage")
  On ne l'utilise pas. Le token JWT est stocké dans localStorage pour persister entre les pages et les sessions.
  sessionStorage aurait été plus sécurisé (effacé à la fermeture du navigateur) mais on a choisi localStorage pour éviter de reconnecter le host à chaque fois.

localStorage (se prononce "local-storage")
  Stocke le token JWT après connexion (localStorage.setItem('token', data.token)).
  On le lit dans host.js pour autoriser les requêtes API.
  Pourquoi pas sessionStorage ? Voir ci-dessus.
  Pourquoi pas un cookie ? Les cookies ont des contraintes CSRF, besoin de SameSite, et le serveur n'est pas encore configuré pour lire des cookies.

Authorization (se prononce "autorization")
  En-tête HTTP envoyé avec chaque requête API : Authorization: Bearer <token>.
  Le serveur vérifie ce token pour identifier l'utilisateur et protéger les routes (ex: CRUD quiz).
  Pourquoi pas une session classique ? Parce qu'on utilise une architecture SPA + API REST, pas de rendu serveur.

Socket.io (se prononce "soket i-o")
  Librairie de communication en temps réel (WebSocket avec fallback).
  Utilisée pour : jeu en direct (création de partie, rejoindre, questions, réponses, classement).
  Pourquoi pas du polling HTTP ? Le polling serait lent et coûteux (requête toutes les secondes).
  Socket.io permet l'émission d'événements en temps réel avec un ping maintenu entre le client et le serveur.

renderQuizList (se prononce "rendeur quiz liste")
  Fonction qui affiche la liste des quiz dans le dashboard host.
  Récupère le tableau allQuizzes, génère des .quiz-card en HTML via boucle forEach.
  Met à jour le compteur #quiz-count.
  Pourquoi pas un template HTML statique ? Parce que la liste est dynamique (création/suppression) et dépend des données du serveur.

showQuestion (se prononce "cho question")
  Fonction qui affiche la question en cours pendant la partie.
  Met à jour #display-question, les 4 réponses (#answer-0-text à #answer-3-text), le timer.
  Passe du lobby à l'écran de jeu (cache lobbySection, montre gameSection).
  Gère le compte à rebours avec setInterval (décrémente, change la couleur à ≤5s).

validate (se prononce "valideute")
  Fonction appelée au clic sur le bouton de soumission (connexion/inscription).
  Vérifie : email non vide + contient @, password non vide, confirmation match.
  Ajoute la classe .input-error sur les champs invalides et .visible sur les messages d'erreur.
  Pourquoi pas une validation HTML5 (required, pattern) ? On l'utilise en complément, mais la validation JS permet des messages personnalisés et évite le submit.

CRUD (se prononce "croude" — pour Create, Read, Update, Delete)
  Les 4 opérations de base sur les données : créer, lire, mettre à jour, supprimer.
  Dans le projet : création de quiz (POST), affichage (GET), modification (PUT), suppression (DELETE).

API REST (se prononce "api reste")
  Architecture qui expose des ressources via des URLs et méthodes HTTP.
  Exemple : /api/quizzes en GET pour lister, POST pour créer, DELETE pour supprimer.

JWT (se prononce "jé-wa-té" — JSON Web Token)
  Token contenant des infos cryptées (ex: email). Stocké dans localStorage.
  Envoyé dans l'en-tête Authorization pour prouver l'identité.

SPA (se prononce "s-p-a" — Single Page Application)
  Application sur une seule page HTML. Le contenu change sans rechargement.
  Ici : host.html, joueur.html sont des mini-SPA. Les sections sont cachées/montrées.


Pourquoi as-tu choisi de gérer tes états de cette manière ?

classList.add('hidden') / classList.remove('hidden')
  Chaque "écran" est une section dans le HTML (selection-section, game-section, lobby-section, ...).
  On cache/montre les sections en ajoutant/enlevant la classe CSS .hidden { display: none !important; }.
  Pourquoi pas un framework (React, Vue) ? Le projet est petit, pas besoin d'un virtual DOM.
  Pourquoi pas du display: none en inline ? Parce que les classes sont plus maintenables et réutilisables.
  Pourquoi pas du routing (changer d'URL) ? Le jeu est un flux linéaire, pas de navigation indépendante. 
  Pourquoi pas innerHTML pour tout le contenu ? On garde la structure HTML de base dans le fichier, on cache/montre les sections au lieu de tout regénérer.


Comment fonctionne ton socket ?

Au chargement, les clients (player.js et host.js) se connectent au serveur Socket.io.
  const socket = io(); // se connecte automatiquement au même hôte

Le host émet (socket.emit) :
  - host_create_game   → crée une partie, reçoit game_created avec le PIN
  - host_start_game    → démarre la partie, les joueurs reçoivent game_started
  - host_next_question  → passe à la question suivante
  - host_time_up       → le temps est écoulé
  - host_show_leaderboard → affiche le classement

Le host écoute (socket.on) :
  - game_created         → reçoit le PIN, affiche le lobby
  - player_list_update   → liste des joueurs mise à jour
  - all_players_answered → tous les joueurs ont répondu
  - game_started         → la partie commence
  - next_question        → question suivante
  - answer_reveal        → correction de la question (bonne réponse, pourcentages)
  - show_leaderboard     → classement après chaque question
  - quiz_ended           → podium final
  - update_response_count → compteur de réponses en direct

Le joueur émet (player.js) :
  - join_game    → rejoint avec PIN + pseudo
  - submit_answer → envoie sa réponse

Le joueur écoute :
  - game_started     → début de partie
  - new_question     → reçoit une question
  - answer_result    → résultat de sa réponse (bonne/mauvaise)
  - leaderboard_update → classement après chaque question
  - quiz_ended       → fin du quiz
  - error            → erreur (ex: mauvais PIN)

PIN (se prononce "pine" — Personal Identification Number)
  Code à 4 chiffres affiché par le host. Les joueurs le tapent pour rejoindre la partie.
  Il est généré par le serveur et unique pour chaque partie en cours.


Autres fonctions et variables importantes :

showToast(msg, type) (se dit "cho tost")
  Utilisée dans tous les fichiers JS. Affiche une notification temporaire en haut.
  Le paramètre type peut être "success" (vert) ou "error" (rouge).
  Ajoute la classe .visible pour montrer le toast.

hideToast() (se dit "haïde tost")
  Cache le toast en retirant la classe .visible.

clearErrors() (se dit "clir erreurs")
  Appelée par validate(). Enlève les classes .input-error et .visible sur les messages d'erreur.
  Présente dans login.js et register.js.

renderQuestionList() (se dit "rendeur question liste")
  Dans host.js. Affiche la liste des questions dans le sidebar pendant la création du quiz.
  Chaque question est un .q-chip avec son numéro, un bouton supprimer et un clic pour modifier.

editQuestion(index) (se dit "édit question")
  Dans host.js. Charge une question existante dans le formulaire pour la modifier.
  Change le texte du bouton en "Mettre à jour la question".

deleteQuestion(index) (se dit "délit question")
  Dans host.js. Supprime une question du tableau temporaire tempQuestions.

resetQuestionForm() (se dit "rizet question form")
  Dans host.js. Vide le formulaire d'ajout de question et réinitialise currentEditingIndex à null.

openEditQuiz(id) (se dit "open édit quiz")
  Dans host.js. Charge un quiz existant (titre + questions) dans le formulaire de création pour le modifier.

selectQuiz(quizId) (se dit "séléct quiz")
  Dans host.js. Appelée quand le host clique sur "Lancer la partie".
  Émet host_create_game via Socket.io avec le quiz sélectionné.

deleteQuizFromServer(id) (se dit "délit quiz from serveur")
  Dans host.js. Supprime un quiz via une requête DELETE à /api/quizzes/{id}.
  Affiche une confirmation avant de supprimer.

getUserEmail() (se dit "gète utilisateur imèl")
  Dans host.js. Appelée au chargement. Récupère l'email du host via /api/auth/me.
  Si le token est invalide (401), redirige vers connexion.html.

loadQuizzes() (se dit "lod quiz")
  Dans host.js. Appelée au chargement et après chaque création/modification/suppression.
  Récupère tous les quiz via GET /api/quizzes et appelle renderQuizList().

Variables globales host.js :
  currentPin      → le PIN de la partie en cours (4 chiffres)
  allQuizzes      → tableau de tous les quiz chargés depuis le serveur
  timerInterval   → ID du setInterval pour le compte à rebours
  isLastQuestion  → booléen, true si c'est la dernière question du quiz
  editingQuizId   → ID du quiz en cours d'édition (null si nouvelle création)
  tempQuestions   → tableau temporaire des questions pendant la création/édition
  currentEditingIndex → index de la question en cours d'édition dans tempQuestions
  token           → le JWT récupéré depuis localStorage

Variables player.js :
  btnValider      → bouton "Valider" sur rejoindre.html
  inputPin        → champ du code PIN
  inputPseudo     → champ du pseudo
  playerId        → stocké dans sessionStorage ('kahoot_playerId')
  savedPin        → PIN stocké après connexion ('kahoot_pin')
  savedPseudo     → pseudo stocké après connexion ('kahoot_pseudo')

sessionStorage (player.js)
  Utilisé par le joueur pour garder le PIN, le pseudo et le playerId.
  Contrairement à localStorage (pour le host), le joueur n'a pas besoin de persistance longue durée.
  Clés utilisées : kahoot_pin, kahoot_pseudo, kahoot_playerId.

setTimeout / setInterval
  setTimeout → utilisé pour les redirections après succès (ex: 600ms après connexion).
  setInterval → utilisé dans showQuestion pour le compte à rebours du timer (décrémente toutes les 1s).
  Pourquoi pas requestAnimationFrame ? Pas besoin, le timer n'a pas besoin d'être synchronisé avec le rendu graphique.

addEventListener (se dit "add évènement listener")
  Méthode standard pour attacher des événements (click, keydown).
  Exemple : btn.addEventListener('click', () => { ... });
  Pourquoi pas onclick directement ? addEventListener permet plusieurs handlers sur le même élément.

socket.emit (se dit "soket imit")
  Envoie un événement au serveur avec des données.
  Exemple : socket.emit('host_create_game', quiz); // le host crée une partie

socket.on (se dit "soket one")
  Écoute un événement venant du serveur.
  Exemple : socket.on('game_created', (data) => { ... });

setItem / getItem / removeItem
  Méthodes de localStorage et sessionStorage pour stocker/lire/supprimer des données.
  Exemple : localStorage.setItem('token', data.token);
  Exemple : sessionStorage.getItem('kahoot_pin');
  Exemple : sessionStorage.removeItem('kahoot_pin');

data-answer (se dit "data anne-sœur")
  Attribut HTML personnalisé. Utilisé sur les boutons .pad-btn pour stocker l'index de réponse (0 à 3).
  Récupéré via button.getAttribute('data-answer').

querySelector / querySelectorAll (se dit "query sélécteur")
  Méthodes pour sélectionner des éléments DOM.
  querySelector → un seul élément (ex: #screen-waiting h2)
  querySelectorAll → plusieurs éléments (ex: .pad-btn)

endsWith (se dit "endz ouith")
  Méthode string. Vérifie si une chaîne finit par une valeur donnée.
  Exemple : window.location.pathname.endsWith('joueur.html') → true si on est sur la page joueur.

classList.contains (se dit "class liste conten-t")
  Vérifie si un élément a une classe CSS. Utilisé dans les guards avant de modifier des éléments.
